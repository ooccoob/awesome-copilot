# 子代理驱动开发

通过为每个任务派遣新的子代理并在每个任务后进行代码审查来执行计划。

**核心原则：** 每个任务的新子代理 + 任务间审查 = 高质量，快速迭代

## 概述

**vs. 执行计划（并行会话）：**
- 相同会话（无上下文切换）
- 每个任务的新子代理（无上下文污染）
- 每个任务后代码审查（及早发现问题）
- 更快迭代（任务间无需人工介入）

**何时使用：**
- 留在此会话中
- 任务大多独立
- 想要带质量门的持续进展

**何时不使用：**
- 需要先审查计划（使用executing-plans）
- 任务紧密耦合（手动执行更好）
- 计划需要修订（先头脑风暴）

## 过程

### 1. 加载计划

阅读计划文件，用所有任务创建TodoWrite。

### 2. 用子代理执行任务

对于每个任务：

**派遣新子代理：**
```
Task工具（通用）：
  description："实现任务N：[任务名称]"
  prompt: |
    您正在实现来自[plan-file]的任务N。

    仔细阅读该任务。您的工作是：
    1. 准确实现任务指定的内容
    2. 编写测试（如果任务说要用TDD）
    3. 验证实现有效
    4. 提交您的工作
    5. 报告回来

    工作目录：[directory]

    报告：您实现了什么，测试了什么，测试结果，更改的文件，任何问题
```

**子代理返回**工作摘要。

### 3. 审查子代理的工作

**派遣code-reviewer子代理：**
```
Task工具（superpowers:code-reviewer）：
  使用requesting-code-review/code-reviewer.md中的模板

  WHAT_WAS_IMPLEMENTED：[来自子代理的报告]
  PLAN_OR_REQUIREMENTS：来自[plan-file]的任务N
  BASE_SHA：[任务前的提交]
  HEAD_SHA：[当前提交]
  DESCRIPTION：[任务摘要]
```

**代码审查器返回：** 优点，问题（关键/重要/次要），评估

### 4. 应用审查反馈

**如果发现问题：**
- 立即修复关键问题
- 在下一个任务前修复重要问题
- 记录次要问题

**如果需要派遣后续子代理：**
```
"修复代码审查的问题：[问题列表]"
```

### 5. 标记完成，下一个任务

- 在TodoWrite中标记任务为完成
- 移动到下一个任务
- 重复步骤2-5

### 6. 最终审查

所有任务完成后，派遣最终code-reviewer：
- 审查整个实现
- 检查所有计划要求满足
- 验证整体架构

### 7. 完成开发

最终审查通过后：
- 宣布："我正在使用完成开发分支技能来完成这项工作。"
- **必需子技能：** 使用superpowers:finishing-a-development-branch
- 遵循该技能验证测试，展示选项，执行选择

## 示例工作流

```
您：我正在使用子代理驱动开发来执行这个计划。

[加载计划，创建TodoWrite]

任务1：钩子安装脚本

[派遣实现子代理]
子代理：用测试实现了install-hook，5/5通过

[获取git SHA，派遣code-reviewer]
审查者：优点：良好测试覆盖。问题：无。就绪。

[标记任务1完成]

任务2：恢复模式

[派遣实现子代理]
子代理：添加了verify/repair，8/8测试通过

[派遣code-reviewer]
审查者：优点：扎实。问题（重要）：缺少进度报告

[派遣修复子代理]
修复子代理：每100个对话添加进度

[验证修复，标记任务2完成]

...

[所有任务后]
[派遣最终code-reviewer]
最终审查者：所有要求满足，准备好合并

完成！
```

## 优势

**vs. 手动执行：**
- 子代理自然遵循TDD
- 每个任务新上下文（无混淆）
- 并行安全（子代理不干扰）

**vs. 执行计划：**
- 相同会话（无交接）
- 持续进展（无等待）
- 审查检查点自动

**成本：**
- 更多子代理调用
- 但及早发现问题（比之后调试更便宜）

## 红旗

**绝不：**
- 跳过任务间代码审查
- 在未修复关键问题下继续
- 并行派遣多个实现子代理（冲突）
- 不阅读计划任务就实现

**如果子代理任务失败：**
- 派遣带具体指令的修复子代理
- 不要尝试手动修复（上下文污染）

## 集成

**必需工作流技能：**
- **writing-plans** - 必需：创建此技能执行的计划
- **requesting-code-review** - 必需：每个任务后审查（见步骤3）
- **finishing-a-development-branch** - 必需：所有任务后完成开发（见步骤7）

**子代理必须使用：**
- **test-driven-development** - 子代理为每个任务遵循TDD

**替代工作流：**
- **executing-plans** - 用于并行会话而不是相同会话执行

参见code-reviewer模板：requesting-code-review/code-reviewer.md
