# 测试反模式

## 概述

测试必须验证真实行为，不是模拟行为。模拟是隔离的手段，不是被测试的东西。

**核心原则：** 测试代码做什么，不是模拟做什么。

**遵循严格TDD防止这些反模式。**

## 铁律

```
1. 绝不测试模拟行为
2. 绝不向生产类添加仅测试方法
3. 绝不在理解依赖关系前模拟
```

## 反模式1：测试模拟行为

**违规：**
```typescript
// ❌ 坏：测试模拟存在
test('渲染侧边栏', () => {
  render(<Page />);
  expect(screen.getByTestId('sidebar-mock')).toBeInTheDocument();
});
```

**为什么错误：**
- 您在验证模拟有效，不是组件有效
- 模拟存在时测试通过，不存在时失败
- 对真实行为什么都不告诉

**您的人类合作伙伴的纠正：** "我们在测试模拟的行为吗？"

**修复：**
```typescript
// ✅ 好：测试真实组件或不模拟它
test('渲染侧边栏', () => {
  render(<Page />);  // 不模拟侧边栏
  expect(screen.getByRole('navigation')).toBeInTheDocument();
});

// 或者如果侧边栏必须为隔离而模拟：
// 不要断言模拟 - 测试存在侧边栏时页面的行为
```

### 闸门功能

```
在任何模拟元素断言之前：
  问："我在测试真实组件行为还是仅模拟存在？"

  如果测试模拟存在：
    停止 - 删除断言或取消模拟组件

  而是测试真实行为
```

## 反模式2：生产中的仅测试方法

**违规：**
```typescript
// ❌ 坏：destroy()仅在测试中使用
class Session {
  async destroy() {  // 看起来像生产API！
    await this._workspaceManager?.destroyWorkspace(this.id);
    // ... 清理
  }
}

// 在测试中
afterEach(() => session.destroy());
```

**为什么错误：**
- 生产类被仅测试代码污染
- 如果在生产中意外调用危险
- 违反YAGNI和关注点分离
- 混淆对象生命周期和实体生命周期

**修复：**
```typescript
// ✅ 好：测试工具处理测试清理
// Session没有destroy() - 在生产中无状态

// 在test-utils/
export async function cleanupSession(session: Session) {
  const workspace = session.getWorkspaceInfo();
  if (workspace) {
    await workspaceManager.destroyWorkspace(workspace.id);
  }
}

// 在测试中
afterEach(() => cleanupSession(session));
```

### 闸门功能

```
在向生产类添加任何方法之前：
  问："这仅被测试使用吗？"

  如果是：
    停止 - 不要添加
    而是把它放在测试工具中

  问："这个类拥有此资源的生命周期吗？"

  如果否：
    停止 - 错误的类放此方法
```

## 反模式3：不理解就模拟

**违规：**
```typescript
// ❌ 坏：模拟破坏测试逻辑
test('检测重复服务器', () => {
  // 模拟阻止测试依赖的配置写入！
  vi.mock('ToolCatalog', () => ({
    discoverAndCacheTools: vi.fn().mockResolvedValue(undefined)
  }));

  await addServer(config);
  await addServer(config);  // 应该抛出 - 但不会！
});
```

**为什么错误：**
- 模拟方法有测试依赖的副作用（写入配置）
- 过度模拟"为了安全"破坏实际行为
- 测试因错误原因通过或神秘失败

**修复：**
```typescript
// ✅ 好：在正确级别模拟
test('检测重复服务器', () => {
  // 模拟慢部分，保留测试需要的行为
  vi.mock('MCPServerManager'); // 仅模拟慢服务器启动

  await addServer(config);  // 配置写入
  await addServer(config);  // 重复检测 ✓
});
```

### 闸门功能

```
在模拟任何方法之前：
  停止 - 还不要模拟

  1. 问："真实方法有什么副作用？"
  2. 问："这个测试依赖那些副作用中的任何吗？"
  3. 问："我完全理解这个测试需要什么吗？"

  如果依赖副作用：
    在较低级别模拟（实际慢/外部操作）
    或使用保留必要行为的测试替身
    不是测试依赖的高级方法

  如果不确定测试依赖什么：
    首先用真实实现运行测试
    观察实际需要发生什么
    然后在正确级别添加最小模拟

  红旗：
    - "我会模拟这个为了安全"
    - "这可能慢，最好模拟它"
    - 不理解依赖链就模拟
```

## 反模式4：不完整模拟

**违规：**
```typescript
// ❌ 坏：部分模拟 - 只有您认为需要的字段
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' }
  // 缺少：下游代码使用的元数据
};

// 稍后：当代码访问response.metadata.requestId时破坏
```

**为什么错误：**
- **部分模拟隐藏结构假设** - 您只模拟您知道的字段
- **下游代码可能依赖您未包含的字段** - 静默失败
- **测试通过但集成失败** - 模拟不完整，真实API完整
- **错误信心** - 测试对真实行为什么都证明不了

**铁律：** 模拟现实中存在的完整数据结构，不只是您的直接测试使用的字段。

**修复：**
```typescript
// ✅ 好：镜像真实API完整性
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' },
  metadata: { requestId: 'req-789', timestamp: 1234567890 }
  // 真实API返回的所有字段
};
```

### 闸门功能

```
在创建模拟响应之前：
  检查："真实API响应包含什么字段？"

  操作：
    1. 检查文档/示例中的实际API响应
    2. 包含系统可能在下游消费的所有字段
    3. 验证模拟完全匹配真实响应模式

  关键：
    如果您在创建模拟，您必须理解整个结构
    当代码依赖省略字段时部分模拟静默失败

  如果不确定：包含所有记录的字段
```

## 反模式5：集成测试事后考虑

**违规：**
```
✅ 实现完成
❌ 没有写测试
"准备好测试"
```

**为什么错误：**
- 测试是实现的一部分，不是可选的后续
- TDD本应捕获这个
- 没有测试不能声称完成

**修复：**
```
TDD循环：
1. 写失败测试
2. 实现通过
3. 重构
4. 然后声称完成
```

## 何时模拟变得太复杂

**警告标志：**
- 模拟设置比测试逻辑长
- 模拟一切使测试通过
- 模拟缺少真实组件有的方法
- 模拟更改时测试破坏

**您的人类合作伙伴的问题：** "我们需要在这里使用模拟吗？"

**考虑：** 带真实组件的集成测试通常比复杂模拟更简单

## TDD防止这些反模式

**为什么TDD有帮助：**
1. **先写测试** → 强迫您思考您实际在测试什么
2. **观察它失败** → 确认测试测试真实行为，不是模拟
3. **最小实现** → 没有仅测试方法渗入
4. **真实依赖** → 在模拟前您看到测试实际需要什么

**如果您在测试模拟行为，您违反了TDD** - 您在观察测试对真实代码失败之前添加了模拟。

## 快速参考

| 反模式 | 修复 |
|--------------|-----|
| 断言模拟元素 | 测试真实组件或取消模拟 |
| 生产中的仅测试方法 | 移动到测试工具 |
| 不理解就模拟 | 首先理解依赖，最小模拟 |
| 不完整模拟 | 完全镜像真实API |
| 测试事后考虑 | TDD - 测试先行 |
| 过度复杂模拟 | 考虑集成测试 |

## 红旗

- 断言检查`*-mock`测试ID
- 仅在测试文件中调用的方法
- 模拟设置 > 测试的50%
- 当您移除模拟时测试失败
- 无法解释为什么需要模拟
- "为了安全"模拟

## 底线

**模拟是隔离的工具，不是测试的东西。**

如果TDD揭示您在测试模拟行为，您错了。

修复：测试真实行为或质疑为什么根本要模拟。
