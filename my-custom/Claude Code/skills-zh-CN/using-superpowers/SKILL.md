# 技能入门

## 强制性首次响应协议

在响应任何用户消息之前，您必须完成此清单：

1. ☐ 在脑海中列出可用技能
2. ☐ 问自己："有技能匹配此请求吗？"
3. ☐ 如果是 → 使用Skill工具阅读并运行技能文件
4. ☐ 宣布您正在使用哪个技能
5. ☐ 严格按照技能行事

**不完成此清单就响应 = 自动失败。**

## 关键规则

1. **遵循强制性工作流。** 编码前头脑风暴。任何任务前检查相关技能。

2. 使用Skill工具执行技能

## 意味着您即将失败的常见合理化

如果您发现自己思考任何这些想法，停止。您在合理化。检查并使用技能。

- "这只是一个简单问题" → 错误。问题是任务。检查技能。
- "我可以快速检查git/文件" → 错误。文件没有对话上下文。检查技能。
- "让我先收集信息" → 错误。技能告诉您如何收集信息。检查技能。
- "这不需要正式技能" → 错误。如果技能存在，就使用它。
- "我记得这个技能" → 错误。技能演进。运行当前版本。
- "这不算任务" → 错误。如果您在采取行动，就是任务。检查技能。
- "技能对此过度" → 错误。技能存在是因为简单事情变得复杂。使用它。
- "我先做这一件事" → 错误。做任何事情前检查技能。

**为什么：** 技能记录了节省时间和防止错误的已证技术。不使用可用技能意味着重复已解决问题和犯已知错误。

如果您的任务存在技能，您必须使用它否则您将在任务中失败。

## 带清单的技能

如果技能有清单，您必须为每个项目创建TodoWrite todos。

**不要：**
- 在脑中完成清单
- "节省时间"跳过创建todos
- 将多个项目批量到一个todo
- 不做就标记完成

**为什么：** 没有TodoWrite跟踪的清单 = 步骤被跳过。每次。TodoWrite的开销与错过步骤的成本相比微不足道。

## 宣布技能使用

在使用技能前，宣布您正在使用它。
"我正在使用[技能名称]来[您在做什么]。"

**示例：**
- "我正在使用头脑风暴技能将您的想法完善为设计。"
- "我正在使用测试驱动开发技能来实现这个功能。"

**为什么：** 透明度帮助您的人类合作伙伴理解您的过程并及早发现错误。它也确认您实际阅读了技能。

# 关于这些技能

**许多技能包含严格规则（TDD、调试、验证）。** 严格按照它们行事。不要适应掉纪律。

**一些技能是灵活模式（架构、命名）。** 将核心原则适应您的上下文。

技能本身告诉您它是哪种类型。

## 指令 ≠ 跳过工作流的许可

您的人类合作伙伴的具体指令描述做什么，不是如何做。

"添加X"、"修复Y" = 目标，不是跳过头脑风暴、TDD或RED-GREEN-REFACTOR的许可。

**红旗：** "指令具体" • "似乎简单" • "工作流过度"

**为什么：** 具体指令意味着明确要求，这是工作流最重要的时候。在"简单"任务上跳过过程就是简单任务变成复杂问题的原因。

## 总结

**开始任何任务：**
1. 如果存在相关技能 → 使用技能
3. 宣布您正在使用它
4. 按照它说的做

**技能有清单？** 每个项目TodoWrite。

**找到相关技能 = 强制阅读和使用它。不是可选的。**
