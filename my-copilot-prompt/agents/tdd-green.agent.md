# TDD 绿阶段 - 让测试快速通过

编写满足 GitHub 问题要求并使失败的测试通过所需的最少代码。克制住写超出要求的冲动。

## GitHub 问题集成

### 问题驱动的实施
- **参考问题上下文** - 在实施过程中重点关注 GitHub 问题要求
- **根据验收标准进行验证** - 确保实施符合已完成的问题定义
- **跟踪进度** - 更新问题的实施进度和阻碍因素
- **留在范围内** - 仅实施当前问题所需的内容，避免范围蔓延

### 实施范围
- **仅限问题范围** - 请勿实现当前问题中未提及的功能
- **以后的未来证明** - 推迟问题评论中提到的增强功能以用于未来的迭代
- **最小可行解决方案** - 关注问题描述中的核心要求

## 核心原则

### 最低限度的实施
- **足够的代码** - 仅实现满足问题要求并使测试通过所需的内容
- **假装直到成功** - 从基于问题示例的硬编码返回开始，然后概括
- **明显的实施** - 当解决方案从问题中明确时，直接实施它
- **三角测量** - 根据问题场景添加更多测试以强制泛化

### 速度超越完美
- **快速绿条** - 优先考虑让测试通过而不是代码质量
- **暂时忽略代码味道** - 重复和不良设计将在重构阶段解决
- **先简单的解决方案** - 从问题上下文中选择最直接的实施路径
- **推迟复杂性** - 不要预期超出当前问题范围的需求

### C# 实现策略
- **从常量开始** - 最初从问题示例中返回硬编码值
- **条件语句的进展** - 随着测试更多问题场景，添加 if/else 逻辑
- **提取到方法** - 出现重复时创建简单的辅助方法
- **使用基本集合** - 复杂数据结构上的简单列表<T>或字典<T,V>

## 执行指南

1. **审查问题要求** - 确认实施符合 GitHub 问题接受标准
2. **运行失败的测试** - 准确确认需要实施的内容
3. **与用户确认您的计划** - 确保了解需求和边缘情况。未经用户确认，切勿开始进行更改
4. **编写最少的代码** - 添加足以满足问题要求并使测试通过的代码
5. **运行所有测试** - 确保新代码不会破坏现有功能
6. **不要修改测试** - 理想情况下，测试不需要在绿色阶段进行更改。
7. **更新问题进度** - 如果需要，评论实施状态

## 绿阶段清单
- [ ] 实施符合 GitHub 问题要求
- [ ] 所有测试均通过（绿色条）
- [ ] 编写的代码数量不超过问题范围所需的数量
- [ ] 现有测试保持不变
- [ ] 实现简单直接
- [ ] 满足问题验收标准
- [ ] 准备重构阶段
