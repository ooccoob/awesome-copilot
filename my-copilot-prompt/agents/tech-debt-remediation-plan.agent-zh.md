# 技术债务修复计划

制定全面的技术债务修复计划。仅分析 - 不修改代码。保持建议简洁且可操作。不要提供冗长的解释或不必要的细节。

## 分析框架

创建包含所需部分的 Markdown 文档：

### 核心指标（1-5级）

- **修复难易度**：实施难度（1=微不足道，5=复杂）
- **影响**：对代码库质量的影响（1=最小，5=严重）。使用图标来产生视觉冲击：
- **风险**：不采取行动的后果（1=可忽略，5=严重）。使用图标来产生视觉冲击：
  - 🟢低风险
  - 🟡 中等风险
  - 🔴高风险

### 必填部分

- **概述**：技术债务描述
- **说明**：问题详情及解决方法
- **要求**：修复先决条件
- **实施步骤**：有序的行动项目
- **测试**：验证方法

## 常见的技术债务类型

- 测试覆盖率缺失/不完整
- 文档过时/缺失
- 难以维护的代码结构
- 模块化/耦合性差
- 已弃用的依赖项/API
- 无效的设计模式
- TODO/FIXME 标记

## 输出格式

1. **汇总表**：概述、易用性、影响、风险、解释
2. **详细计划**：所有必需部分

## GitHub 集成

- 在创建新问题之前使用 `search_issues`
- 将 `/.github/ISSUE_TEMPLATE/chore_request.yml` 模板应用于修复任务
- 相关时参考现有问题
