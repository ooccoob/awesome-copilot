# 产品需求文件（PRD）

## 概述

设计全面的生产级产品需求文档 (PRD)，以弥合业务愿景和技术执行之间的差距。这项技能适用于现代软件系统，确保明确定义需求。

## 何时使用

在以下情况下使用此技能：

- 开始新产品或功能开发周期
- 将模糊的想法转化为具体的技术规范
- 定义人工智能驱动功能的要求
- 利益相关者需要项目范围的统一“事实来源”
- 用户要求“编写 PRD”、“记录需求”或“规划功能”

---

## 操作流程

### 第一阶段：发现（采访）

在编写 PRD 的一行之前，您**必须**询问用户以填补知识空白。不要假设上下文。

**询问：**

- **核心问题**：我们为什么现在要构建这个？
- **成功指标**：我们如何知道它有效？
- **限制**：预算、技术堆栈或截止日期？

### 第 2 阶段：分析和范围界定

综合用户的输入。识别依赖性和隐藏的复杂性。

- 绘制**用户流程**。
- 定义**非目标**以保护时间表。

### 第三阶段：技术起草

使用下面的**严格 PRD 架构**生成文档。

---

## 珠三角质量标准

### 要求质量

使用具体的、可衡量的标准。避免“快速”、“简单”或“直观”。

```diff
# Vague (BAD)
- The search should be fast and return relevant results.
- The UI must look modern and be easy to use.

# Concrete (GOOD)
+ The search must return results within 200ms for a 10k record dataset.
+ The search algorithm must achieve >= 85% Precision@10 in benchmark evals.
+ The UI must follow the 'Vercel/Next.js' design system and achieve 100% Lighthouse Accessibility score.
```

## 严格的 PRD 架构

您**必须**遵循以下确切的输出结构：

### 1. 执行摘要

- **问题陈述**：针对痛点的 1-2 句话。
- **建议的解决方案**：关于修复的 1-2 句话。
- **成功标准**：3-5 个可衡量的 KPI。

### 2. 用户体验和功能

- **用户角色**：这是给谁的？
- **用户故事**：`As a [user], I want to [action] so that [benefit].`
- **验收标准**：每个故事的“完成”定义的项目符号列表。
- **非目标**：我们不构建什么？

### 3. AI系统要求（如适用）

- **工具要求**：需要哪些工具和 API？
- **评估策略**：如何衡量输出质量和准确性。

### 4. 技术规格

- **架构概述**：数据流和组件交互。
- **集成点**：API、数据库和身份验证。
- **安全和隐私**：数据处理和合规性。

### 5. 风险与路线图

- **分阶段推出**：MVP -> v1.1 -> v2.0。
- **技术风险**：延迟、成本或依赖性故障。

## 实施指南

### 做（总是）

- **定义测试**：对于 AI 系统，指定如何测试和验证输出质量。
- **迭代**：提出草稿并请求对特定部分的反馈。

### 不要（避免）

- **跳过发现**：在没有首先提出至少 2 个澄清问题的情况下，切勿编写 PRD。
- **幻觉约束**：如果用户未指定技术堆栈，请询问或将其标记为 `TBD`。

## 示例：智能搜索系统

### 1. 执行摘要

**问题**：用户很难在大量存储库中找到特定的文档片段。
**解决方案**：智能搜索系统，提供带有来源引用的直接答案。
**成功**：

- 搜索时间减少 50%。
- 引用准确率 >= 95%。

### 2. 用户故事

- **故事**：作为一名开发人员，我想问自然语言问题，这样我就不必猜测关键字。
- **交流**：
  - 支持多轮澄清。
  - 使用“复制”按钮返回代码块。

### 3. 人工智能系统架构

- **所需工具**：`codesearch`、`grep`、`webfetch`。

### 4. 评价

- **基准**：使用 50 个常见的开发人员问题进行测试。
- **通过率**：90% 必须符合预期引用。
