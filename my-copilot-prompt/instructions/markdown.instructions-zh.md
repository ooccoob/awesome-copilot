## Markdown 内容规则

验证器中强制执行以下降价内容规则：

1. **标题**：使用适当的标题级别（H2、H3 等）来构建内容。不要使用 H1 标题，因为它将根据标题生成。
2. **列表**：使用项目符号或编号列表作为列表。确保适当的缩进和间距。
3. **代码块**：使用受隔离的代码块作为代码片段。指定语法突出显示的语言。
4. **链接**：对链接使用正确的 Markdown 语法。确保链接有效且可访问。
5. **图像**：对图像使用正确的 Markdown 语法。包含替代文本以方便访问。
6. **表格**：使用降价表格来存储表格数据。确保正确的格式和对齐方式。
7. **行长度**：将行长度限制为 400 个字符以提高可读性。
8. **空格**：使用适当的空格来分隔各部分并提高可读性。
9. **前言**：在文件开头包含 YAML 前言以及所需的元数据字段。

## 格式和结构

请遵循以下准则来格式化和构建 Markdown 内容：

- **标题**：对于 H2 使用 `##`，对于 H3 使用 `###`。确保以分层方式使用标题。如果内容包含H4，建议重构，更强烈推荐H5。
- **列表**：使用 `-` 作为项目符号点，使用 `1.` 作为编号列表。用两个空格缩进嵌套列表。
- **代码块**：使用三个反引号 (`) to create fenced code blocks. Specify the language after the opening backticks for syntax highlighting (e.g., `csharp)。
- **链接**：使用 `[link text](URL)` 作为链接。确保链接文本具有描述性并且 URL 有效。
- **图像**：使用 `![alt text](imageURL)` 作为图像。在替代文本中包含图像的简短描述。
- **表**：使用 `|` 创建表。确保列正确对齐并包含标题。
- **行长**：在 80 个字符处换行以提高可读性。对于长段落使用软换行符。
- **空白**：使用空行分隔各部分并提高可读性。避免过多的空白。

## 验证要求

确保符合以下验证要求：

- **前言**：在 YAML 前言中包含以下字段：

  - `post_title`：帖子的标题。
  - `author1`：该帖子的主要作者。
  - `post_slug`：帖子的 URL slug。
  - `microsoft_alias`：作者的 Microsoft 别名。
  - `featured_image`：特色图像的 URL。
  - `categories`：帖子的类别。这些类别必须来自 /categories.txt 中的列表。
  - `tags`：帖子的标签。
  - `ai_note`：表明在创建帖子时是否使用了人工智能。
  - `summary`：帖子的简短摘要。尽可能根据内容推荐摘要。
  - `post_date`：帖子的发布日期。

- **内容规则**：确保内容遵循上面指定的 Markdown 内容规则。
- **格式**：确保内容的格式和结构符合指南。
- **验证**：运行验证工具来检查是否符合规则和指南。
