# C# 文档最佳实践

- 应使用 XML 注释记录公共成员。
- 我们也鼓励记录内部成员，特别是当它们很复杂或不言自明时。

## 所有 API 的指南

- 使用 `<summary>` 提供类型或成员功能的简短一句话描述。以现在时的第三人称动词开始摘要。
- 使用 `<remarks>` 获取其他信息，其中可以包括实现细节、使用说明或任何其他相关上下文。
- 将 `<see langword>` 用于特定于语言的关键字，例如 `null`、`true`、`false`、`int`、`bool` 等。
- 使用 `<c>` 作为内联代码片段。
- 使用 `<example>` 作为有关如何使用该成员的用法示例。
  - 使用 `<code>` 作为代码块。 `<code>` 标签应放置在 `<example>` 标签内。使用 `language` 属性添加代码示例的语言，例如 `<code language="csharp">`。
- 使用 `<see cref>` 内联引用其他类型或成员（在句子中）。
- 在在线文档的“另请参阅”部分中，使用 `<seealso>` 对其他类型或成员进行独立（不在句子中）引用。
- 使用 `<inheritdoc/>` 从基类或接口继承文档。
  - 除非行为发生重大变化，否则您应该记录差异。

## 方法

- 使用 `<param>` 来描述方法参数。
  - 描述应该是不指定数据类型的名词短语。
  - 从一篇介绍性文章开始。
  - 如果参数是标志枚举，则以“指定...的枚举值的按位组合”开始描述。
  - 如果参数是非标志枚举，则以“指定...的枚举值之一”开始描述。
  - 如果参数是布尔值，则措辞应采用“`<see langword="true" />` to ...；否则，`<see langword="false" />`.”的形式。
  - 如果参数是“out”参数，则措辞应采用“当此方法返回时，包含...。此参数被视为未初始化。”的形式。
- 使用 `<paramref>` 引用文档中的参数名称。
- 使用 `<typeparam>` 描述泛型类型或方法中的类型参数。
- 使用 `<typeparamref>` 引用文档中的类型参数。
- 使用 `<returns>` 描述该方法返回的内容。
  - 描述应该是不指定数据类型的名词短语。
  - 从一篇介绍性文章开始。
  - 如果返回类型为布尔值，则措辞应采用“`<see langword="true" />` if ...；否则，`<see langword="false" />`.”的形式。

## 构造函数

- 摘要措辞应该是“初始化<Class>类[或结构]的新实例。”。

## 属性

- `<summary>` 应以以下内容开头：
  - “获取或设置...”用于读写属性。
  - “获取...”表示只读属性。
  - 对于返回布尔值的属性，“获取[或设置]一个指示是否...的值”。
- 使用 `<value>` 描述属性的值。
  - 描述应该是不指定数据类型的名词短语。
  - 如果该属性有默认值，请将其添加到单独的句子中，例如“默认值为 `<see langword="false" />`”。
  - 如果值类型是布尔型，则措辞应采用“`<see langword="true" />` if ...；否则，`<see langword="false" />`。默认为...”的形式。

## 例外情况

- 使用 `<exception cref>` 记录构造函数、属性、索引器、方法、运算符和事件引发的异常。
- 记录成员直接引发的所有异常。
- 对于嵌套成员引发的异常，仅记录用户最有可能遇到的异常。
- 异常的描述描述了抛出该异常的条件。
  - 省略句子开头的“如果...抛出”或“如果...”。只需直接说明条件即可，例如“访问消息队列 API 时发生错误”。
